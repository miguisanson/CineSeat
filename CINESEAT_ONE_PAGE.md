# CineSeat: One-Page Application and Module Reflection

**Application:** CineSeat | **Platform:** iOS 17.5 | **Language/UI:** Swift 5 and UIKit | **Bundle ID:** `SevenSeven.CineSeat` | **Architecture:** MVVM | **Data source:** Local movie data, online poster images, and device persistence

## Application Overview

CineSeat is a cinema booking app based on a Figma wireframe. A user can search and filter movies, sort each movie filter by highest or lowest rating, view online poster images, choose a scheduled showing with an assigned cinema, select available seats, review prices, confirm a booking after login, view the cinema map again from booking details, cancel saved bookings, and manage a local profile through account creation, login, editing, logout, and restored sessions. The main storyboard contains three tab-based navigation flows: **Movies**, **Bookings**, and **Profile**.

**Main flow:** Movies -> Movie Detail -> Scheduled Showing -> Seat Selection -> Booking Summary -> Confirmation -> Booking History. Profile -> Create Account/Login -> View/Edit Profile -> Logout.

## Lecture Module Reflection

| Module | How CineSeat Applies the Concepts |
| --- | --- |
| **1: Swift Fundamentals** | Uses `let`/`var`, `String`, `Int`, `Double`, `Bool`, arrays for movies/bookings/seats, dictionaries for Keychain queries, and `Set<String>` for unique selected/reserved seats. Operators calculate totals and compare status/rating values. `if`, `guard`, `switch`, ranges, and `for` loops control filtering, validation, and seat generation. Functions and properties use Swift camelCase. Optionals handle missing selections, saved files, and sessions. `throw`, `try`, `do-catch`, and `LocalizedError` propagate readable account/storage errors. `Movie`, `Booking`, and `UserProfile` are value-type structs; view controllers, stores, repositories, and view models are reference-type classes. Persistence protocols separate contracts from implementations. `Codable`, `JSONEncoder`, and `JSONDecoder` serialize data. ARC is respected through `weak` outlets and `[weak self]` closure captures. No third-party framework is required. |
| **2: UIKit and MVVM** | **Model:** movie, cinema, fixed movie showing, booking, draft, profile, status, and validation data. **View:** storyboard scenes, labels, buttons, text fields, alerts, table views, reusable cells, scroll views, and stack views. **ViewModel:** movie search/category filtering, rating sorting, fixed showing selection, booking filtering, seat validation/sorting/pricing, login, registration, and profile updates. UIKit lifecycle methods include `viewDidLoad`, `viewWillAppear`, and `deinit`. Storyboard outlets/actions connect the Movies and Bookings screens; Profile navigation is storyboard-connected and its content is built with UIKit. Auto Layout anchors and vertical/horizontal `UIStackView` layouts support different iPhone sizes. `UITableView` and reusable cells display movies and bookings. Button target-actions provide interaction; alerts confirm destructive actions and booked-seat taps. `UITextFieldDelegate` handles Return-key navigation, while `UIScrollView` keeps account and booking forms usable with compact screens and keyboards. |
| **3: Intermediate Swift** | Closures are passed to `filter`, `map`, `sorted`, `first(where:)`, alert actions, and notification handlers. Standard generic collections such as `Array`, `Set`, and `Dictionary` provide type safety. Protocol-oriented design appears in `BookingPersisting`, `ProfilePersisting`, and `PasswordStoring`, which also allow in-memory test implementations. Functional operations filter movies/bookings, map seat rows, flatten text-field trees, and sort selected seats. Errors propagate from repositories/Keychain through services and view models to UI `catch` blocks. Advanced `Result`, GCD, `OperationQueue`, `async/await`, and actors are not currently necessary because operations are small and local; they become relevant when remote APIs or heavier processing are introduced. |
| **4: Networking and Dependencies** | Poster images can load from bundled `PosterImages` files for offline use, then fall back to URL/cache loading when needed. Online image loading still uses `URLSession`, updates the UI on the main thread, and saves downloads into a `PosterCache` folder with `FileManager`. Movie list data comes from `SampleDataJson.json` through `MockMovieAPIClient`, which follows the `MovieFetching` protocol and can be replaced by a real GET API client later. The app does not need POST, API tokens, retry logic, CocoaPods, Swift Package Manager packages, or Alamofire for the current scope. |
| **5: Data Persistence and Security** | `UserDefaults` stores simple preferences and flags: first launch, selected movie category, cancelled-booking visibility, and the signed-in profile ID. `FileManager` locates the Documents and Caches directories; `JSONDecoder` reads bundled seed data from `SampleDataJson.json`, `JSONEncoder` writes `bookings.json` and `profiles.json`, and poster files are available from the bundled `PosterImages` folder for offline use. CRUD is represented by creating bookings/accounts, fetching saved arrays, updating booking status/profile details, filtering with closure predicates, and deleting or replacing Keychain password records when required. New installs start with no booking history until a signed-in user confirms a booking. Passwords are read and written with the iOS Keychain instead of plain JSON or UserDefaults. Core Data is not used; CineSeat uses Codable JSON with FileManager because the current data set is small. A Keychain Access Group is not configured because data is not shared with another app or extension. |
| **6: MVVM, Clean Architecture, and DI** | The current structure separates **Presentation** (`View` and `ViewModel`), **Domain** (`DependencyProtocols` and `UseCases`), **Design** (`UIComponents` and `AppFactory`), and **Model/Data** (`Models` and persistence). View models receive protocol-based dependencies, so a future real API fetch can replace the current `MockMovieAPIClient` without changing the UI. `AppFactory` shows the Factory Pattern by building view models and flow screens from shared dependencies. Singletons remain only for app-wide services such as preferences, booking storage, authentication, and the factory composition root. |

## Quality and Scope

The project includes unit tests for movie filtering and rating sort, scheduled cinema rotation, seat rules, totals, cinema prices, booking persistence, preferences, account validation, authentication, duplicate accounts, and Keychain operations. UI tests cover movie-to-seat navigation, the signed-out bookings state, and the complete create-account/logout/login flow. The app has been built for iPhone 15 and iPhone SE simulators. Current limitations are deliberate: no backend account service, live cinema API, real payment, cloud synchronization, or real-time seat locking.
