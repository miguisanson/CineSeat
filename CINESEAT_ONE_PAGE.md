# CineSeat: One-Page Application and Module Reflection

**Application:** CineSeat | **Platform:** iOS 17.5 | **Language/UI:** Swift 5 and UIKit | **Bundle ID:** `SevenSeven.CineSeat` | **Architecture:** MVVM | **Data source:** JSON seed data, plist settings/seat layouts, poster files, and device persistence

## Application Overview

CineSeat is a showing and ticket booking app based on a Figma wireframe. A user starts in **Showings** and filters one shared list by Movies, Concerts, or Seminars. Search works across each category, movie results still use All/Now Playing/Coming Soon and rating order, and a location menu filters movies by assigned cinemas or events by venue. Movies use scheduled cinemas and seat maps. Concerts and seminars use live relative dates, scheduled times, venue coordinates, per-ticket prices, and a quantity selector without seat selection. Both flows require login before confirmation, save into the same FileManager-backed `bookings.json`, support ticket ownership/sharing, schedule local reminders, show booking history/details, and support cancellation. The Locations tab switches between cinema and event-venue MapKit pins; pin callouts open assigned movies or events. The app also includes local settings and a profile flow with account creation, login, editing, logout, and restored sessions. The main storyboard contains four tab-based navigation flows: **Showings**, **Bookings**, **Locations**, and **Profile**.

**Main flow:** Showings -> Category/Cinema or Venue Filter -> Movie or Event Detail. Movie -> Date/Time -> Seat Selection -> Summary -> Confirmation. Concert/Seminar -> Date/Time/Venue Map -> Ticket Quantity -> Summary -> Confirmation. Locations -> Cinema/Event Venue Pins -> Assigned Showings. Profile -> Create Account/Login -> View/Edit Profile -> Settings -> Logout.

## Lecture Module Reflection

| Module | How CineSeat Applies the Concepts |
| --- | --- |
| **1: Swift Fundamentals** | Uses `let`/`var`, `String`, `Int`, `Double`, `Bool`, arrays for movies/events/bookings/seats, dictionaries for Keychain queries, and `Set<String>` for unique selected/reserved seats. Operators calculate totals and compare status/rating values. `if`, `guard`, `switch`, ranges, and `for` loops control filtering, validation, and seat generation. Functions and properties use Swift camelCase. Optionals handle missing selections, saved files, poster URLs, and sessions. `throw`, `try`, `do-catch`, and `LocalizedError` propagate readable account/storage errors. `Movie`, `EventListing`, `Booking`, and `UserProfile` are value-type structs; view controllers, stores, repositories, and view models are reference-type classes. Persistence protocols separate contracts from implementations. `Codable`, `JSONEncoder`, and `JSONDecoder` serialize data. ARC is respected through `weak` outlets and `[weak self]` closure captures. No third-party framework is required. |
| **2: UIKit and MVVM** | **Model:** movies, events, cinemas, event venues, nested movie/event schedules, generic `BookingItem`, movie/event drafts, profiles, settings, and seat layouts. **View:** storyboard scenes, buttons, segmented controls, menus, search bars, alerts, table views, stack/scroll views, reusable MapKit previews, interactive map pin callouts, `UIStepper` ticket quantity, and the custom `SeatMapView`. **ViewModel:** Showings filters, movie rating/status filters, event date/time/quantity/pricing, cinema and event-venue map categories, booking filtering, seat rules, settings, and account flows. UIKit lifecycle methods include `viewDidLoad`, `viewWillAppear`, and `deinit`. Auto Layout anchors and stack views support different iPhone sizes. Movie seating uses vertical rows and horizontal seat buttons, while events deliberately skip seat selection and calculate totals from quantity. |
| **3: Intermediate Swift** | Closures are passed to `filter`, `map`, `sorted`, `first(where:)`, alert actions, and notification handlers. Standard generic collections such as `Array`, `Set`, and `Dictionary` provide type safety. Protocol-oriented design appears in `MovieFetching`, `EventFetching`, `BookingPersisting`, `ProfilePersisting`, `AppSettingsManaging`, and `PasswordStoring`, which also allow mock or file-based implementations. Functional operations filter movies/events/bookings, map seat rows, flatten text-field trees, and sort selected seats. Errors propagate from repositories/Keychain through services and view models to UI `catch` blocks. Local notifications are scheduled through `UserNotifications` for booking reminders without using APNs. Advanced `Result`, GCD, `OperationQueue`, `async/await`, and actors are not currently necessary because operations are small and local; they become relevant when remote APIs or heavier processing are introduced. |
| **4: Networking and Dependencies** | Poster images load from bundled files or URL/cache loading with `URLSession` and `FileManager`. Split resources now include `Movies.json`, `Concerts.json`, `Seminars.json`, `Cinemas.json`, `Showings.json`, `EventVenues.json`, `EventShowings.json`, `Bookings.json`, and `ProfileAccounts.json`. `daysFromToday` values are mapped into current calendar dates. Protocol-backed movie, event, movie-showing, and event-showing clients can later be replaced by real GET APIs. Apple's native `MapKit` supplies cinema and event venue pins, while UIKit `UIDatePicker` handles schedule dates without CocoaPods. |
| **5: Data Persistence and Security** | `UserDefaults` stores small flags and the signed-in profile ID. Settings and seat layouts use plist files through `PropertyListEncoder`, `PropertyListDecoder`, and `FileManager`. Bundled JSON provides read-only movies, events, venues, and schedules; `JSONFileWriter` saves user-created movie and event purchases together in `bookings.json`, while profiles use `profiles.json`. `BookingItem` preserves whether a record contains a movie or event, and its custom decoder remains compatible with older movie-only booking files. `TicketAssignment` stores either seat IDs or numbered event ticket IDs for ownership and sharing. Passwords remain in Keychain. Core Data is not used. |
| **6: MVVM, Clean Architecture, and DI** | Presentation remains in `View`/`ViewModel`, contracts and use cases remain in `Domain`, composition remains in `AppFactory`, and models/persistence remain separate. `EventScheduleViewModel` owns event date/time/quantity/pricing, while views only render and route. `FetchMovieShowingsUseCase`, `FetchEventShowingsUseCase`, `ConfirmBookingUseCase`, and `ConfirmEventBookingUseCase` use protocol-backed clients and the shared booking manager. `BookingItem` lets one repository store movie and event records safely. The factory injects schedule, authentication, confirmation, cancellation, sharing, settings, and notification dependencies. Shared constants and custom views keep UI behavior consistent. |

## Quality and Scope

The project includes unit tests for Showings filters, movie and event schedules, live relative dates, venue assignments, map categories, event ticket quantities/totals, movie seats, JSON booking persistence, sharing, settings, authentication, and Keychain operations. UI tests cover movie seat booking, concert ticket quantity/summary, signed-out bookings, and account flows. Current limitations are deliberate: no backend, live event API, payment gateway, cloud synchronization, real-time capacity locking, or turn-by-turn GPS navigation.
