# TicketPlease: One-Page Application and Module Reflection

**Application:** TicketPlease | **Platform:** iOS 17.5 | **Language/UI:** Swift 5 and UIKit | **Bundle ID:** `SevenSeven.CineSeat` | **Architecture:** MVVM | **Data source:** JSON seed data, plist settings/seat layouts, poster files, and device persistence

## Application Overview

TicketPlease is a showing and ticket booking app based on a Figma wireframe. A user starts in **Showings**, where separate Movies, Concerts, and Seminars cards open their respective pages. All three pages consistently provide a title, search, All/Now Showing/Coming Soon status controls, ascending/descending rating order, and a cinema or venue filter. Movies use scheduled cinemas and seat maps. Concerts and seminars use live relative dates, scheduled times, venue coordinates, per-ticket prices, and a quantity selector without seat selection. Every detail page links to a separate Reviews screen backed by 60 casual seeded reviews. Online and TicketPlease user ratings remain separate; sorting/display uses the online score when available and the app average as fallback. Both booking flows require login before confirmation, save into the same FileManager-backed `bookings.json`, support ticket ownership/sharing, schedule local reminders, show booking history/details, and support cancellation. The Locations tab switches between cinema and event-venue MapKit pins; pin callouts open assigned showings. The app also includes local settings and a profile flow with account creation, login, editing, logout, and restored sessions.

**Main flow:** Showings -> Movies, Concerts, or Seminars card -> Search/Status/Rating/Location Filters -> Detail -> Reviews. Movie -> Date/Time -> Seat Selection -> Summary -> Confirmation. Concert/Seminar -> Date/Time/Venue Map -> Ticket Quantity -> Summary -> Confirmation. Locations -> Cinema/Event Venue Pins -> Assigned Showings. Profile -> Create Account/Login -> View/Edit Profile -> Settings -> Logout.

## Lecture Module Reflection

| Module | How TicketPlease Applies the Concepts |
| --- | --- |
| **1: Swift Fundamentals** | Uses `let`/`var`, `String`, `Int`, `Double`, `Bool`, arrays for movies/events/reviews/bookings/seats, dictionaries for Keychain queries, and `Set<String>` for unique selected/reserved seats. Operators calculate totals and app review averages. `if`, `guard`, `switch`, ranges, and `for` loops control filtering, validation, and seat generation. Functions and properties use Swift camelCase. Optionals represent unavailable online ratings, missing selections, saved files, poster URLs, and sessions. `Movie`, `Concert`, `Seminar`, `Review`, `Booking`, and `UserProfile` are value-type structs, while `EventListing` is an enum that safely carries either ticketed type. `Codable`, `JSONEncoder`, and `JSONDecoder` serialize data. ARC is respected through `weak` outlets and `[weak self]` closure captures. No third-party framework is required. |
| **2: UIKit and MVVM** | **Model:** separate movie, concert, seminar, and review models; cinemas; event venues; nested schedules; bookings; profiles; settings; and seat layouts. **View:** a category-card landing page; uniform list titles, search bars, status segmented controls, rating buttons, and location menus; separate review table screen/cell; alerts; stack/scroll views; MapKit previews; `UIStepper`; and `SeatMapView`. **ViewModel:** separate movie/concert/seminar filtering, a review rating summary, shared ticket schedule logic, map categories, booking filtering, seat rules, settings, and account flows. Auto Layout anchors and stack views support different iPhone sizes. |
| **3: Intermediate Swift** | Closures are passed to `filter`, `map`, `sorted`, `first(where:)`, `reduce`, alert actions, and notification handlers. Standard generic collections provide type safety. Protocol-oriented design appears in `MovieFetching`, `EventFetching`, `ReviewFetching`, `BookingPersisting`, `ProfilePersisting`, and settings/password contracts. Functional operations filter content/reviews/bookings, calculate review averages, map seat rows, and sort ratings. Errors propagate through services/view models. Local notifications use `UserNotifications` without APNs. |
| **4: Networking and Dependencies** | Poster images load from bundled files or URL/cache loading with `URLSession` and `FileManager`. Split resources include movie/concert/seminar/location/schedule/account/booking JSON plus `Reviews.json`. Protocol-backed movie, event, showing, and review clients can later be replaced by real GET APIs. Apple's native `MapKit` supplies pins, while UIKit `UIDatePicker` handles dates without CocoaPods. |
| **5: Data Persistence and Security** | `UserDefaults` stores small flags and the signed-in profile ID. Settings and seat layouts use plist files through `PropertyListEncoder`, `PropertyListDecoder`, and `FileManager`. Bundled JSON provides read-only content and 60 review records; `BundledReviewRepository` decodes `Reviews.json` separately. `JSONFileWriter` saves purchases in `bookings.json`, profiles use `profiles.json`, and passwords remain in Keychain. Core Data is not used. Seed reviews are demo data and are not user-editable yet. |
| **6: MVVM, Clean Architecture, and DI** | Presentation remains in `View`/`ViewModel`, contracts/use cases remain in `Domain`, composition remains in `AppFactory`, and models/persistence remain separate. Reviews have dedicated folders across all layers. `ReviewFetching` hides the JSON repository, `DefaultFetchReviewsUseCase` filters reviews and calculates the app average, and `ReviewsViewModel` formats the screen. Movie/concert/seminar view models use the same rating rule while retaining separate files and category logic. The factory injects reviews alongside schedules, authentication, booking, sharing, settings, and notifications. |

## Quality and Scope

The project includes unit tests for review JSON coverage, online/app rating fallback, uniform category filters/sorts, schedules, venue assignments, maps, ticket totals, seats, booking persistence, sharing, settings, authentication, and Keychain operations. UI tests cover review routing from movie/concert details plus booking and account flows. Current limitations are deliberate: reviews are bundled demo records with no posting/backend yet; there is also no payment gateway, cloud synchronization, real-time capacity locking, or turn-by-turn GPS navigation.
