# CineSeat: One-Page Application and Module Reflection

**Application:** CineSeat | **Platform:** iOS 17.5 | **Language/UI:** Swift 5 and UIKit | **Bundle ID:** `SevenSeven.CineSeat` | **Architecture:** MVVM | **Data source:** JSON seed data, plist settings/seat layouts, poster files, and device persistence

## Application Overview

CineSeat is a showing and cinema booking app based on a Figma wireframe. A user starts in **Showings**, chooses Movies, Concerts, or Seminars, then can browse each category with poster images from bundled files or online URL/cache loading. Movies keep the full cinema flow: search/filter movies, sort by rating, choose a scheduled showing with an assigned cinema, select seats from that cinema's own plist layout, review prices, confirm a booking after login, share one seat ticket to another local account by email, receive local reminder notifications, view the seat map again from booking details, and cancel saved bookings. The app also includes a dedicated MapKit cinema location tab where map pins open cinema details and assigned schedules, local settings, and a profile flow with account creation, login, editing, logout, and restored sessions. The main storyboard contains four tab-based navigation flows: **Showings**, **Bookings**, **Locations**, and **Profile**.

**Main flow:** Showings -> Movies -> Movie Detail -> Scheduled Showing -> Seat Selection -> Booking Summary -> Confirmation -> Booking History. Showings -> Concerts/Seminars -> Event Detail. Locations -> Cinema Map -> Cinema Details -> Assigned Movie. Profile -> Create Account/Login -> View/Edit Profile -> Settings -> Logout.

## Lecture Module Reflection

| Module | How CineSeat Applies the Concepts |
| --- | --- |
| **1: Swift Fundamentals** | Uses `let`/`var`, `String`, `Int`, `Double`, `Bool`, arrays for movies/events/bookings/seats, dictionaries for Keychain queries, and `Set<String>` for unique selected/reserved seats. Operators calculate totals and compare status/rating values. `if`, `guard`, `switch`, ranges, and `for` loops control filtering, validation, and seat generation. Functions and properties use Swift camelCase. Optionals handle missing selections, saved files, poster URLs, and sessions. `throw`, `try`, `do-catch`, and `LocalizedError` propagate readable account/storage errors. `Movie`, `EventListing`, `Booking`, and `UserProfile` are value-type structs; view controllers, stores, repositories, and view models are reference-type classes. Persistence protocols separate contracts from implementations. `Codable`, `JSONEncoder`, and `JSONDecoder` serialize data. ARC is respected through `weak` outlets and `[weak self]` closure captures. No third-party framework is required. |
| **2: UIKit and MVVM** | **Model:** movie, event, cinema, fixed movie showing, booking, draft, profile, status, validation data, settings, and per-cinema seat layouts. **View:** storyboard scenes, labels, buttons, text fields, alerts, table views, reusable cells, scroll views, stack views, settings controls, event cards, MapKit pin callouts, and the custom `SeatMapView`. **ViewModel:** Showings category selection, event browsing/search, cinema detail schedule filtering, movie search/category filtering, rating sorting, fixed showing selection, booking filtering, seat validation/sorting/pricing, settings toggles, login, registration, and profile updates. UIKit lifecycle methods include `viewDidLoad`, `viewWillAppear`, and `deinit`. Storyboard outlets/actions connect the movie and booking screens; the Showings landing page, event lists, Locations map, cinema detail, and Profile content are built with UIKit. Auto Layout anchors and vertical/horizontal `UIStackView` layouts support different iPhone sizes. The seating UI uses custom UIKit views under `CustomViews/Seating`, built from vertical rows and horizontal seat buttons, so cinemas can have different rows, aisle positions, reserved seats, and unavailable seats. `UITableView` and reusable cells display movies, events, and bookings. Button target-actions provide interaction; alerts confirm destructive actions and booked-seat taps. `UITextFieldDelegate` handles Return-key navigation, while `UIScrollView` keeps account, settings, and booking forms usable with compact screens and keyboards. |
| **3: Intermediate Swift** | Closures are passed to `filter`, `map`, `sorted`, `first(where:)`, alert actions, and notification handlers. Standard generic collections such as `Array`, `Set`, and `Dictionary` provide type safety. Protocol-oriented design appears in `MovieFetching`, `EventFetching`, `BookingPersisting`, `ProfilePersisting`, `AppSettingsManaging`, and `PasswordStoring`, which also allow mock or file-based implementations. Functional operations filter movies/events/bookings, map seat rows, flatten text-field trees, and sort selected seats. Errors propagate from repositories/Keychain through services and view models to UI `catch` blocks. Local notifications are scheduled through `UserNotifications` for booking reminders without using APNs. Advanced `Result`, GCD, `OperationQueue`, `async/await`, and actors are not currently necessary because operations are small and local; they become relevant when remote APIs or heavier processing are introduced. |
| **4: Networking and Dependencies** | Poster images can load from bundled `PosterImages` files for offline movie use, then fall back to URL/cache loading when needed. Online image loading still uses `URLSession`, updates the UI on the main thread, and saves downloads into a `PosterCache` folder with `FileManager`; concerts and seminars use online Unsplash poster/example URLs through the same loader. Showing data comes from split bundled JSON files (`Movies.json`, `Concerts.json`, `Seminars.json`, `Cinemas.json`, `Showings.json`, an empty launch `Bookings.json`, and `ProfileAccounts.json`) through `SeedDataStore`, `MockMovieAPIClient`, and `MockEventAPIClient`, which follow protocol-based fetch contracts and can be replaced by real GET API clients later. Cinema locations use Apple's native `MapKit`, so no pod is needed for the map. The app keeps UIKit `UIDatePicker` for booking dates because the requirement is a simple native calendar, so CocoaPods or a third-party calendar would add dependency setup without meaningful benefit right now. |
| **5: Data Persistence and Security** | `UserDefaults` stores tiny flags such as first launch, selected movie category, and signed-in profile ID. App settings now use `DefaultAppSettings.plist` and `CineSeatSettings.plist` through `PropertyListEncoder`, `PropertyListDecoder`, and `FileManager`; the Settings screen can toggle cancelled bookings, booking reminders, and demo notifications. Seat layouts now come from `SeatLayouts.plist`, which acts as a local editable database for cinema rows, aisle positions, reserved seats, and unavailable seats. `JSONFileReader` and `JSONDecoder` read bundled seed data from separate category JSON files, while `JSONFileWriter` and `JSONEncoder` write user-created bookings to the app container `bookings.json` and user-created profiles to `profiles.json`. Bookings now save `TicketAssignment` rows so each purchased seat can belong to one account and be shared to another account by email. CRUD is represented by creating bookings/accounts, fetching saved arrays, updating booking status/profile details/settings/ticket ownership, filtering with closure predicates, and deleting or replacing Keychain password records when required. Passwords are read and written with the iOS Keychain instead of plain JSON, plist, or UserDefaults. Core Data is not used because the current data is small and explainable with Codable plus FileManager. |
| **6: MVVM, Clean Architecture, and DI** | The current structure separates **Presentation** (`View` and `ViewModel`), **Domain** (`DependencyProtocols` and `UseCases`), **Design** (`CustomViews`, UI constants, and `AppFactory`), and **Model/Data** (`Models` and `Persistence`). `Models` is split into Showings, Movies, Events, Booking, Profile, and Settings, while persistence is split into Shared, SeedData, Booking, Profile, Settings, SeatLayouts, and Notifications. View models receive protocol-based dependencies, so a future real API fetch or different settings store can replace the current implementations without changing the UI. `FetchEventsUseCase`, `ConfirmBookingUseCase`, `TransferTicketUseCase`, and the booking/account protocols keep browsing, checkout, and ticket sharing away from concrete stores. `AppFactory` shows the Factory Pattern by building view models and flow screens from shared dependencies. UI constants such as fonts, spacing, button heights, radii, poster sizes, and seat sizes are centralized in `CineSeatUIConstants.swift` to make visual changes scalable. Singletons remain only for app-wide services such as settings, preferences, booking storage, authentication, and the factory composition root. |

## Quality and Scope

The project includes unit tests for showings/event browsing, movie filtering and rating sort, split seed JSON resources, scheduled cinema rotation, plist seat layouts, seat rules, totals, cinema prices, booking persistence, ticket sharing, plist settings, preferences, account validation, authentication, duplicate accounts, and Keychain operations. UI tests cover Showings-to-movie-to-seat navigation, the signed-out bookings state, settings preferences, and the complete create-account/logout/login flow. The app has been built for iPhone 15 and iPhone SE simulators. Current limitations are deliberate: no backend account service, live cinema/event API, real payment, cloud synchronization, or real-time seat locking.
