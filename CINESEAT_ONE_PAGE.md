# CineSeat: One-Page Application and Module Reflection

**Application:** CineSeat | **Platform:** iOS 17.5 | **Language/UI:** Swift 5 and UIKit | **Bundle ID:** `SevenSeven.CineSeat` | **Architecture:** MVVM | **Data source:** JSON seed data, plist settings/seat layouts, poster files, and device persistence

## Application Overview

CineSeat is a cinema booking app based on a Figma wireframe. A user can search and filter movies, sort each movie filter by highest or lowest rating, view bundled/offline poster images with online fallback, choose a scheduled showing with an assigned cinema, select seats from that cinema's own plist layout, review prices, confirm a booking after login, receive local reminder notifications, view the cinema map again from booking details, cancel saved bookings, edit local settings, and manage a local profile through account creation, login, editing, logout, and restored sessions. The main storyboard contains three tab-based navigation flows: **Movies**, **Bookings**, and **Profile**.

**Main flow:** Movies -> Movie Detail -> Scheduled Showing -> Seat Selection -> Booking Summary -> Confirmation -> Booking History. Profile -> Create Account/Login -> View/Edit Profile -> Settings -> Logout.

## Lecture Module Reflection

| Module | How CineSeat Applies the Concepts |
| --- | --- |
| **1: Swift Fundamentals** | Uses `let`/`var`, `String`, `Int`, `Double`, `Bool`, arrays for movies/bookings/seats, dictionaries for Keychain queries, and `Set<String>` for unique selected/reserved seats. Operators calculate totals and compare status/rating values. `if`, `guard`, `switch`, ranges, and `for` loops control filtering, validation, and seat generation. Functions and properties use Swift camelCase. Optionals handle missing selections, saved files, and sessions. `throw`, `try`, `do-catch`, and `LocalizedError` propagate readable account/storage errors. `Movie`, `Booking`, and `UserProfile` are value-type structs; view controllers, stores, repositories, and view models are reference-type classes. Persistence protocols separate contracts from implementations. `Codable`, `JSONEncoder`, and `JSONDecoder` serialize data. ARC is respected through `weak` outlets and `[weak self]` closure captures. No third-party framework is required. |
| **2: UIKit and MVVM** | **Model:** movie, cinema, fixed movie showing, booking, draft, profile, status, validation data, settings, and per-cinema seat layouts. **View:** storyboard scenes, labels, buttons, text fields, alerts, table views, reusable cells, scroll views, stack views, settings controls, and the custom `SeatMapView`. **ViewModel:** movie search/category filtering, rating sorting, fixed showing selection, booking filtering, seat validation/sorting/pricing, settings toggles, login, registration, and profile updates. UIKit lifecycle methods include `viewDidLoad`, `viewWillAppear`, and `deinit`. Storyboard outlets/actions connect the Movies and Bookings screens; Profile navigation is storyboard-connected and its content is built with UIKit. Auto Layout anchors and vertical/horizontal `UIStackView` layouts support different iPhone sizes. The seating UI uses custom UIKit views under `CustomViews/Seating`, built from vertical rows and horizontal seat buttons, so cinemas can have different rows, aisle positions, reserved seats, and unavailable seats. `UITableView` and reusable cells display movies and bookings. Button target-actions provide interaction; alerts confirm destructive actions and booked-seat taps. `UITextFieldDelegate` handles Return-key navigation, while `UIScrollView` keeps account, settings, and booking forms usable with compact screens and keyboards. |
| **3: Intermediate Swift** | Closures are passed to `filter`, `map`, `sorted`, `first(where:)`, alert actions, and notification handlers. Standard generic collections such as `Array`, `Set`, and `Dictionary` provide type safety. Protocol-oriented design appears in `MovieFetching`, `BookingPersisting`, `ProfilePersisting`, `AppSettingsManaging`, and `PasswordStoring`, which also allow mock or file-based implementations. Functional operations filter movies/bookings, map seat rows, flatten text-field trees, and sort selected seats. Errors propagate from repositories/Keychain through services and view models to UI `catch` blocks. Local notifications are scheduled through `UserNotifications` for booking reminders without using APNs. Advanced `Result`, GCD, `OperationQueue`, `async/await`, and actors are not currently necessary because operations are small and local; they become relevant when remote APIs or heavier processing are introduced. |
| **4: Networking and Dependencies** | Poster images can load from bundled `PosterImages` files for offline use, then fall back to URL/cache loading when needed. Online image loading still uses `URLSession`, updates the UI on the main thread, and saves downloads into a `PosterCache` folder with `FileManager`. Movie list data comes from `SampleDataJson.json` through `MockMovieAPIClient`, which follows the `MovieFetching` protocol and can be replaced by a real GET API client later. The app keeps UIKit `UIDatePicker` for booking dates because the requirement is a simple native calendar, so CocoaPods or a third-party calendar would add dependency setup without meaningful benefit right now. |
| **5: Data Persistence and Security** | `UserDefaults` stores tiny flags such as first launch, selected movie category, and signed-in profile ID. App settings now use `DefaultAppSettings.plist` and `CineSeatSettings.plist` through `PropertyListEncoder`, `PropertyListDecoder`, and `FileManager`; the Settings screen can toggle cancelled bookings, booking reminders, and demo notifications. Seat layouts now come from `SeatLayouts.plist`, which acts as a local editable database for cinema rows, aisle positions, reserved seats, and unavailable seats. `JSONFileReader` and `JSONDecoder` read bundled seed data from `SampleDataJson.json`, while `JSONFileWriter` and `JSONEncoder` write `bookings.json` and `profiles.json`. CRUD is represented by creating bookings/accounts, fetching saved arrays, updating booking status/profile details/settings, filtering with closure predicates, and deleting or replacing Keychain password records when required. Passwords are read and written with the iOS Keychain instead of plain JSON, plist, or UserDefaults. Core Data is not used because the current data is small and explainable with Codable plus FileManager. |
| **6: MVVM, Clean Architecture, and DI** | The current structure separates **Presentation** (`View` and `ViewModel`), **Domain** (`DependencyProtocols` and `UseCases`), **Design** (`CustomViews`, UI constants, and `AppFactory`), and **Model/Data** (`Models` and `Persistence`). `Models` is split into Movies, Booking, Profile, and Settings, while persistence is split into Shared, SampleData, Booking, Profile, Settings, SeatLayouts, and Notifications. View models receive protocol-based dependencies, so a future real API fetch or different settings store can replace the current implementations without changing the UI. `AppFactory` shows the Factory Pattern by building view models and flow screens from shared dependencies. UI constants such as fonts, spacing, button heights, radii, poster sizes, and seat sizes are centralized in `CineSeatUIConstants.swift` to make visual changes scalable. Singletons remain only for app-wide services such as settings, preferences, booking storage, authentication, and the factory composition root. |

## Quality and Scope

The project includes unit tests for movie filtering and rating sort, scheduled cinema rotation, plist seat layouts, seat rules, totals, cinema prices, booking persistence, plist settings, preferences, account validation, authentication, duplicate accounts, and Keychain operations. UI tests cover movie-to-seat navigation, the signed-out bookings state, settings preferences, and the complete create-account/logout/login flow. The app has been built for iPhone 15 and iPhone SE simulators. Current limitations are deliberate: no backend account service, live cinema API, real payment, cloud synchronization, or real-time seat locking.

## Development Change List

- **Day 1 - Initial project setup:** Created the CineSeat iOS project from the wireframe concept, set up the storyboard tab structure, and started the app with UIKit screens for movies, bookings, and profile flow.

- **Day 3 - Booking flow:** Added movie detail, fixed cinema showings, booking summary, peso pricing, seat selection, and booking confirmation. Seat buttons were intentionally separated into clearer beginner-style outlets/actions where useful for presentation.
- **Day 4 - Profile and account flow:** Replaced the old info tab with Profile, then added create account, login, logout, editable profile details, session restore, validation rules, duplicate email checks, and Keychain-backed password storage.
- **Day 5 - Data persistence:** Added FileManager-backed JSON persistence for profiles and bookings, separated JSON reader/writer helpers, moved sample app data into `SampleDataJson.json`, and kept bundled poster images available for offline use.
- **Day 6 - Movie data and posters:** Expanded the movie list to real movie names, added ratings, categories, schedules, offline poster files, URL/cache fallback image loading, and filtering for All, Now Playing, and Coming Soon with ascending/descending rating sort.
- **Day 7 - Cinema rules and seating:** Changed the flow so users select from predetermined movie showings instead of freely choosing any cinema. Added eight cinemas, standard/VIP pricing, different cinema seat layouts, reserved/unavailable seats, and a cinema map view from booking details.
- **Day 8 - Schedule and notification improvements:** Changed schedules into nested date/time structs, extended selectable dates to one week, added multiple showtimes per day, generated current calendar labels, added local booking reminders, and added demo notification support without APNs.
- **Day 9 - Architecture cleanup:** Introduced clearer Clean Architecture folders for Presentation, Domain, Design, Models, and Persistence. Added protocol-based dependency injection, use cases, the app factory, sample data DTOs/mappers, and separated movie, booking, profile, settings, and seat files.
- **Day 10 - Plist settings and seat database:** Added plist persistence for app settings, created the Settings menu under Profile, added plist-backed seat layouts as an editable local database, extracted shared UI constants like fonts/sizes/spacing, improved comments, and updated tests/documentation.
